# Product Constitution

## Purpose

This document defines the principles Rise must follow.

The constitution exists to protect Rise from becoming bloated, overwhelming, guilt-driven, privacy-invasive, or dependent on paid services.

Every future feature should be judged against these rules.

---

## 1. User Data Belongs to the User

Rise must be offline-first.

By default:

* No account required
* No cloud required
* No login required
* No user data uploaded
* No hidden telemetry
* No advertising-based tracking

The user's life data belongs to the user.

Not Rise.

Not advertisers.

Not AI providers.

Not cloud platforms.

---

## 2. Offline First

Rise should work without internet.

Core features must work offline:

* Onboarding
* Dashboard
* Daily check-in
* Health tracking
* Progress tracking
* History
* Backup and restore
* Settings
* Modules

Internet may be useful later, but it must never be required for the core experience.

---

## 3. Progress Over Perfection

Rise should not punish users for missing a day.

The app should support real life.

People get tired.

People fail.

People restart.

People disappear for weeks and come back.

Rise should always help the user continue.

---

## 4. Encourage, Never Shame

Rise must avoid guilt-based design.

Do not say:

> You failed.

Do say:

> Welcome back. Let's continue from where you left off.

If a goal moves backward, Rise should explain the situation clearly and show the quickest path back.

The app should be honest, but never cruel.

---

## 5. Reduce Overwhelm

The user should not be shown too many tasks at once.

One meaningful next step is better than ten random tasks.

Rise should help users focus on what matters now.

The dashboard should feel calm, not crowded.

---

## 6. AI Is Optional

Rise must work fully without AI.

AI should never be required for:

* Tracking
* Progress calculation
* Goal creation
* Milestones
* Reports
* Backup
* Core usage

Future AI should be optional and preferably local-first.

Paid AI APIs should not be required.

---

## 7. Modules, Not Clutter

Rise should grow through modules.

Examples:

* Health
* Mind
* Finance
* Career
* Learning
* Relationships
* Personal Projects
* Hobbies

Users should enable only what they need.

The app should adapt to the user, not force the user into every feature.

---

## 8. Every Feature Needs a Purpose

Before adding a feature, ask:

> Does this help the user know what to do next?

Or:

> Does this help the user see progress?

If the answer is no, the feature should not be added.

---

## 9. The App Should Adapt to the User

Rise should not assume every user has the same life.

The app should adapt based on:

* Goals
* Available time
* Health situation
* Financial situation
* Career stage
* Energy level
* Active modules
* Current progress

The user should not be forced into a generic routine.

---

## 10. Small Steps. Visible Progress.

Rise should make small progress visible.

A 10-minute walk matters.

A 20-minute project session matters.

A small savings deposit matters.

A single honest check-in matters.

Small steps should compound into visible progress.

---

## 11. No Dark Patterns

Rise should not use manipulative design.

Avoid:

* Forced accounts
* Fake urgency
* Shame-based streaks
* Hidden subscriptions
* Data lock-in
* Ads disguised as recommendations
* Making users feel trapped

The app should respect the user.

---

## 12. Backup Must Be Simple

Because Rise is offline-first, backup is important.

Users should be able to export their data easily.

The backup system should be:

* Simple
* Local
* Portable
* Restorable
* Understandable

Users should never feel their life data is trapped inside the app.

---

## 13. Recovery Matters More Than Streaks

Streaks can motivate some users, but they can also make users quit after failure.

Rise should focus on rhythm, recovery, and direction.

If a user misses a day, the app should help them restart with the smallest useful step.

---

## 14. The Dashboard Must Answer One Question

When the user opens Rise, the dashboard should answer:

> What should I do next?

This is the emotional center of the product.

The user should not need to think too much.

They should feel clear.

---

## 15. Progress Must Be Understandable

Graphs and numbers should be simple.

The user should understand:

* Where they started
* Where they are now
* Where they are going
* Whether they are moving closer or further away
* What to do next

Do not show complex data unless it helps the user make a decision.

---

## Final Rule

If a feature makes Rise more complicated but does not make the user's life clearer, do not build it.

Rise exists to create clarity, direction, and visible progress.
